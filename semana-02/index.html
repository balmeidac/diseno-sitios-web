Normas básicas de la web (W3C)

¿Qué es la W3C?

El World Wide Web Consortium (W3C) es la organización internacional que desarrolla estándares para la World Wide Web. Fundada en 1994 por Tim Berners-Lee, el inventor de la web, su misión es garantizar el crecimiento a largo plazo de la web.

Funciones Principales del W3C:

  • Desarrollar estándares: Crear especificaciones técnicas para tecnologías web
  • Validación: Proporcionar herramientas para verificar el cumplimiento de estándares
  • Educación: Promover mejores prácticas y guías de desarrollo
  • Interoperabilidad: Asegurar que diferentes navegadores funcionen consistentemente
  • Accesibilidad: Garantizar que la web sea accesible para todas las personas
Importancia del W3C: Sin estándares unificados, cada navegador podría interpretar el código de manera diferente, creando caos en la experiencia del usuario y dificultando el desarrollo web.

Principales Estándares W3C:

  • HTML: Lenguaje de marcado para estructura de contenido
  • CSS: Hojas de estilo para presentación visual
  • DOM: Document Object Model para manipulación dinámica
  • SVG: Gráficos vectoriales escalables
  • WCAG: Pautas de accesibilidad web
  • ARIA: Aplicaciones internet ricas y accesibles

Estándares y Validación

¿Qué son los Estándares Web?

Los estándares web son especificaciones técnicas que definen cómo deben funcionar las tecnologías web. Estos estándares aseguran que el contenido sea accesible, interoperable y mantenga su calidad a lo largo del tiempo.

Beneficios de Cumplir Estándares:

  • Compatibilidad: Funciona en todos los navegadores modernos
  • Accesibilidad: Disponible para personas con discapacidades
  • Mantenimiento: Código más limpio y fácil de mantener
  • SEO: Mejor posicionamiento en buscadores
  • Futuro: Preparado para tecnologías futuras

Validación

La validación es el proceso de verificar que un documento web cumple con los estándares definidos por el W3C. Es una práctica fundamental para asegurar la calidad del código.

Herramientas de Validación:

  • W3C Markup Validator: Para validar documentos HTML
  • W3C CSS Validator: Para validar hojas de estilo CSS
  • W3C Link Checker: Para verificar enlaces rotos
  • HTML Tidy: Para limpiar y corregir código HTML

Proceso de Validación:

  1. Escribir código siguiendo las especificaciones W3C
  2. Usar herramientas de validación automática
  3. Corregir errores y advertencias encontradas
  4. Verificar que el código sea semánticamente correcto
  5. Probar en diferentes navegadores

Errores Comunes de Validación:

  • Falta de declaración DOCTYPE
  • Etiquetas no cerradas correctamente
  • Atributos obsoletos o no estándar
  • Estructura anidada incorrecta
  • Caracteres especiales no codificados

Accesibilidad Web (WCAG)

La accesibilidad web es la práctica de hacer que los sitios web sean utilizables por todas las personas, incluyendo aquellas con discapacidades visuales, auditivas, motoras o cognitivas. Las Web Content Accessibility Guidelines (WCAG) son el estándar internacional para la accesibilidad web.

¿Por qué es importante la accesibilidad?

  • Inclusión: Permite acceso a personas con discapacidades (15% de la población mundial)
  • Legal: Cumplimiento de leyes y regulaciones en muchos países
  • SEO: Sitios accesibles tienen mejor posicionamiento
  • Usabilidad: Mejora la experiencia para todos los usuarios
  • Etiqueta: Responsabilidad social y empresarial

Tipos de Discapacidades Consideradas:

  • Visuales: Ceguera, baja visión, daltonismo
  • Auditivas: Sordera, hipoacusia
  • Motoras: Dificultades de movimiento, parálisis
  • Cognitivas: Dislexia, autismo, TDAH
  • Técnicas: Conexión lenta, dispositivos antiguos

Tecnologías Asistivas:

  • Lectores de pantalla: JAWS, NVDA, VoiceOver
  • Software de reconocimiento de voz: Dragon NaturallySpeaking
  • Ampliadores de pantalla: ZoomText, Windows Magnifier
  • Teclados adaptados: Teclados en pantalla, switches
  • Braille: Líneas Braille, impresoras Braille

Principios de Usabilidad WCAG

Las WCAG se organizan en cuatro principios fundamentales. Si alguno de estos principios no se cumple, el contenido no es accesible.

1. Perceptible

La información y los componentes de la interfaz deben presentarse de manera que los usuarios puedan percibirlos.

Directrices:

  • Alternativas textuales: Proporcionar texto alternativo para imágenes
  • Medios sincronizados: Subtítulos y descripciones para videos
  • Adaptable: Crear contenido presentable de diferentes formas
  • Distinguible: Facilitar la audición y visión del contenido
<!-- Ejemplo de imagen accesible --> <img src="logo-empresa.png" alt="Logo de TechNova Solutions"> <!-- Video con subtítulos --> <video controls> <source src="video.mp4" type="video/mp4"> <track kind="subtitles" src="subtitulos.vtt" srclang="es"> </video>

2. Operable

Los componentes de la interfaz y la navegación deben ser operables por todos los usuarios.

Directrices:

  • Accesible por teclado: Todas las funciones deben poder usarse con teclado
  • Tiempo suficiente: Dar tiempo suficiente para leer y usar contenido
  • Convulsiones: No diseñar contenido que cause convulsiones
  • Navegable: Ayudar a los usuarios a navegar y encontrar contenido
<!-- Enlace accesible por teclado --> <a href="#contenido" class="skip-link">Saltar al contenido principal</a> <!-- Botón con atajo de teclado --> <button accesskey="s" onclick="buscar()">Buscar [Alt+S]</button>

3. Comprensible

La información y la operación de la interfaz deben ser comprensibles.

Directrices:

  • Legible: El texto debe ser legible y comprensible
  • Predictible: Las páginas web deben funcionar de manera predecible
  • Asistencia: Ayudar a los usuarios a evitar y corregir errores
<!-- Formulario con etiquetas claras --> <label for="email">Correo electrónico:</label> <input type="email" id="email" name="email" required> <!-- Mensajes de error claros --> <span class="error">Por favor, ingrese un correo válido</span>

4. Robusto

El contenido debe ser lo suficientemente robusto para ser interpretado por diferentes tecnologías asistivas.

Directrices:

  • Compatible: Maximizar compatibilidad con tecnologías actuales y futuras
  • Semántico: Usar HTML semántico correctamente
  • ARIA: Usar atributos ARIA cuando sea necesario
<!-- HTML semántico --> <main role="main"> <section aria-labelledby="productos-title"> <h2 id="productos-title">Nuestros Productos</h2> <article aria-label="Producto 1">...</article> </section> </main>

Versiones de WCAG y Niveles de Conformidad

Versiones de WCAG:

  • WCAG 1.0 (1999): Primera versión, 14 directrices
  • WCAG 2.0 (2008): Reestructurado en 4 principios, 12 directrices
  • WCAG 2.1 (2018): Añade 17 criterios de éxito nuevos
  • WCAG 2.2 (2023): Mejoras en accesibilidad móvil y cognitiva

Niveles de Conformidad con las WCAG: A, AA y AAA

Nivel A – Conformidad Básica

Nivel mínimo de accesibilidad. Cumplimiento básico obligatorio.

Requisitos principales:
  • Imágenes con texto alternativo
  • Videos con subtítulos (pregrabados)
  • Contenido accesible por teclado
  • No contenido que cause convulsiones
  • Uso de HTML semántico básico

Nivel AA – Conformidad Intermedia

Nivel recomendado para la mayoría de los sitios web. Equilibrio entre accesibilidad y diseño.

Requisitos adicionales:
  • Contraste de color mínimo 4.5:1
  • Videos con audio descripción
  • Redimensionamiento de texto hasta 200%
  • Funcionamiento completo por teclado
  • Navegación consistente
  • Identificación de errores de formulario

Nivel AAA – Conformidad Avanzada

Nivel máximo de accesibilidad. No es práctico para todo el contenido.

Requisitos adicionales:
  • Contraste de color mínimo 7:1
  • Interpretación por lenguaje de señas
  • Texto ampliado hasta 200% sin pérdida de funcionalidad
  • Interrupción no automática
  • Contexto de ayuda disponible
  • Contenido en lenguaje sencillo
Recomendación: La mayoría de los sitios web deberían aspirar al nivel AA como estándar de accesibilidad. El nivel AAA es ideal para contenido específico pero no es práctico para sitios completos.

Declaración de Conformidad:

Los sitios pueden declarar su nivel de conformidad WCAG mediante:

  • Declaración completa (página completa cumple)
  • Declaración parcial (algunas páginas cumplen)
  • Declaración con conformidad respaldada (verificada por tercera parte)

Ejercicios Resueltos

Ejercicio 2.1: Evaluación de accesibilidad básica en una página web

Enunciado:

Analiza el siguiente fragmento de código HTML y responde: ¿Qué problemas de accesibilidad presenta? ¿Cómo se podrían mejorar para cumplir con las recomendaciones del W3C y las Pautas de Accesibilidad al Contenido en la Web (WCAG)?

<h1>Catálogo de productos</h1> <img src="camisa.jpg"> <a href="pagina2.html">Haz clic aquí</a>

Resolución:

Problemas identificados:
  1. Imagen sin texto alternativo (alt): Las personas con discapacidad visual no sabrán qué representa la imagen.
  2. Enlace no descriptivo: "Haz clic aquí" no es una etiqueta comprensible fuera de contexto, lo que dificulta su uso por parte de lectores de pantalla.
Versión mejorada:
<h1>Catálogo de productos</h1> <img src="camisa.jpg" alt="Camisa de algodón azul para hombre"> <a href="pagina2.html">Ver más detalles del catálogo</a>
Mejoras implementadas:
  • Se añadió atributo alt descriptivo a la imagen
  • Se reemplazó el texto del enlace por uno más descriptivo
  • El código ahora cumple con los principios Perceptible y Comprensible de WCAG

Ejercicio 2.2: Validación de un documento HTML según W3C

Enunciado:

Revisa el siguiente código HTML. Luego, usando los estándares del W3C, identifica los errores que podrían impedir una validación exitosa y corrígelos.

<html> <head> <meta charset=UTF-8> <title>Mi sitio</title> </head> <body> <h1>Bienvenidos</h1> <p>Esta es mi primera página web</body> </html>

Resolución:

Errores encontrados:
  1. Falta la declaración <!DOCTYPE html>.
  2. El atributo charset está mal escrito (debe ir entre comillas).
  3. El elemento <p> no está cerrado.
  4. Código mal indentado.
Código corregido (válido según W3C):
<!DOCTYPE html> <html> <head> <meta charset="UTF-8"> <title>Mi sitio</title> </head> <body> <h1>Bienvenidos</h1> <p>Esta es mi primera página web</p> </body> </html>
Correcciones realizadas:
  • Se añadió la declaración DOCTYPE HTML5
  • Se corrigió el atributo charset con comillas
  • Se cerró correctamente la etiqueta <p>
  • Se mejoró la indentación del código

Ejercicio 2.3: Aplicación práctica de estándares W3C y accesibilidad

Enunciado:

Un desarrollador creó el siguiente formulario. Analiza su conformidad con los estándares del W3C y las pautas de accesibilidad, y luego sugiere una versión mejorada.

<form> Nombre: <input type="text"><br> Correo: <input type="email"><br> <input type="submit" value="Enviar"> </form>

Resolución:

Problemas identificados:
  1. No se están usando etiquetas <label> asociadas a los campos de entrada, lo que afecta a los lectores de pantalla.
  2. No hay atributos name, necesarios para procesar datos del formulario.
  3. No hay agrupación semántica.
Versión mejorada cumpliendo con accesibilidad y estándares:
<form action="procesar.php" method="post"> <fieldset> <legend>Información de contacto</legend> <label for="nombre">Nombre:</label> <input type="text" id="nombre" name="nombre" required><br> <label for="correo">Correo:</label> <input type="email" id="correo" name="correo" required><br> <input type="submit" value="Enviar"> </fieldset> </form>
Mejoras implementadas:
  • Se añadieron etiquetas <label> con atributos for conectados a los inputs
  • Se incluyeron atributos id y name para identificación y procesamiento
  • Se agregó validación con el atributo required
  • Se usó <fieldset> y <legend> para agrupación semántica
  • Se añadieron atributos action y method al formulario
